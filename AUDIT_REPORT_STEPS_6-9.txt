STEP 7: WHOAI AUTHENTICATES REQUEST (API KEY → JWT)
═══════════════════════════════════════════════════════════════════════════════

TOKEN ENDPOINT:
  POST /api/v1/auth/token
  runtime/routers/auth.py:36-98

API KEY FORMATS ACCEPTED:
  1. JSON:  {"api_key": "whoai_sk_..."}
  2. Header: X-API-Key: whoai_sk_...
  3. Bearer: Authorization: Bearer whoai_sk_...

KEY VALIDATION:
  SHA-256 hash of raw key → lookup in Agent.apiKey column
  auth.py:65-67

AGENT STATUS CHECK (Kill-Switch):
  If agent.status != "ACTIVE", return 403 Forbidden
  auth.py:76-80
  ✅ Checked BEFORE token issued

JWT GENERATION:
  HS256 signed with GATEWAY_SECRET env var
  Claims: {"sub": agent_id, "org": org_id, "iat": now, "exp": now + 3600}
  auth.py:83-92
  TTL: 1 hour (3600 seconds)

JWT VALIDATION AT GATEWAY:
  gateway.py:68-81 decodes same JWT with same GATEWAY_SECRET
  Verifies expiration and presence of "sub"/"org" claims

IMPLEMENTATION STATUS: ✅ WORKING
  - Full flow tested (test_auth_token.py:75-92)
  - Suspended agents rejected (test_auth_token.py:65-72)
  - Hash matching validated (test_auth_token.py:15-16)

PRODUCTION CONCERNS:
  🔴 CRITICAL: NO TOKEN REVOCATION
     - Tokens are stateless; no way to revoke if key leaked
     - Leaked key remains valid for full 1-hour TTL
     - Only fix: change GATEWAY_SECRET (breaks all users)
     MITIGATION NEEDED: Add token blacklist table

  🔴 CRITICAL: NO SECRET ROTATION MECHANISM
     - GATEWAY_SECRET has no versioning or rotation ceremony
     - No minimum length validation

  ⚠️  API Key Hash Not Salted
     - Plain SHA-256 (no salt); vulnerable to rainbow tables if DB leaked
     - Defensible only if keys are high-entropy (whoai_sk_... format)
     BETTER: Use bcrypt or PBKDF2

  ⚠️  Bearer Token Parsing Unsafe
     - authorization.split(" ")[1] crashes if header is "Bearer" with no token
     - Should check array length first
